Tier 1 — close the gaps
Start here Evaluation infrastructure ▼

Right now your only quality signal is "tests pass / fail." You have no way to know if agent output is improving, degrading, or whether your reviewer agent is actually doing its job.

  • 1Define 20–30 "golden tasks" — representative inputs with hand-written ideal outputs. Store them in evals/golden/ alongside your existing task format.
  • 2Write an LLM-as-judge prompt: score each agent output on a 1–5 rubric for correctness, completeness, and code quality. Run it after each agent cycle.
  • 3Log scores to evals/results/YYYY-MM-DD.jsonl — one line per task per run. This gives you a time series to track.
  • 4Set a baseline, then gate merges: if mean judge score drops below baseline by >0.3 points, surface a warning before the reviewer agent signs off.

Start here Prompt iteration with feedback loops ▼

Your prompts improve only when you subjectively notice something's wrong. There's no mechanism to confirm a change is actually better.

  • 1Version your agent prompts as AGENT_PROMPT_v1.md, v2.md, etc. Never overwrite — always increment.
  • 2Run both versions against the same 10 golden tasks and compare judge scores. This is your A/B mechanism.
  • 3Add a "prompt diff" section to escape records: note which prompt version was running. After 5–10 failures, look for version-correlated patterns.
  • 4Once you have 4+ prompt versions with score data, try DSPy or PromptBreeder to automate candidate generation.

Layer in Agent memory architecture ▼

PROGRESS.md grows unbounded and every agent reads it from scratch each loop. Agents have no memory of prior attempts within a task run.

  • 1Split PROGRESS.md into a rolling summary file (last 5 items only) plus an archive. Agents read the summary; the archive is for humans and the improvement analyzer.
  • 2Add a per-task scratchpad file (.agent-state/task-{id}.md) that persists across agent loop iterations for that task.
  • 3Define a trim policy: scratchpad files are deleted on task completion; the archive is summarized weekly by the analyzer agent.
  • 4Evaluate whether AthenaMind (or a similar vector store) actually helps vs the scratchpad approach — benchmark retrieval latency and token cost before committing.

Layer in Cross-agent failure aggregation ▼

Escape records are per-task and per-agent. You can't see patterns across agents — e.g., "three agents all failed on Databricks auth this week" only appears in hindsight.

  • 1Standardize your escape record schema: add agent_role, failure_category (auth, context, tool, logic), and task_domain fields to every entry.
  • 2Write a weekly aggregation script that groups escapes by failure_category × agent_role and outputs a Markdown table. Run it every Friday.
  • 3Upgrade the improvement analyzer to trigger on "3+ escapes with the same category in a 48h window" — not just a single escape.
  • 4Feed aggregated failure patterns back into agent prompts as an explicit "known failure modes" section.

Complete Cost and token observability ▼

At scale (10 agents × 8h runs), you have no accounting for which roles are expensive and whether expensive correlates with valuable output.

  • 1Wrap your RTK/API calls with a lightweight logger: capture agent_role, task_id, input_tokens, output_tokens, and timestamp per invocation.
  • 2Build a daily rollup: cost per agent role, cost per task, and cost-per-judge-score-point (your ROI metric).
  • 3Set soft budget alerts: if a single agent role exceeds a per-run token threshold, log a warning to PROGRESS.md.
  • 4Identify the highest-cost role and experiment with a cheaper model tier or shorter context window. Measure the quality delta using your eval infrastructure.

Tier 2 — level up
Level up Adversarial testing and chaos agents ▼

Your eval suite tests happy-path quality. It doesn't test whether your coordination primitives hold under adversarial or malformed inputs — and those are the failures that bite in production.

  • 1Build a "chaos task generator" that produces deliberately broken inputs: malformed PRDs, contradictory requirements, missing lock files, truncated task specs. Add 10–15 of these to your eval suite as a separate adversarial set.
  • 2Define expected failure behavior for each chaos case — the system should fail loudly and gracefully, not silently or corruptly. Score these separately from quality scores.
  • 3Run the adversarial suite on every prompt version change. A prompt improvement that increases quality scores but degrades adversarial resilience is a net regression.
  • 4Periodically introduce real production failures into the chaos set — when something breaks unexpectedly in live runs, write a chaos test that would have caught it.

Level up Dynamic agent spawning ▼

Your agent roles and team topology are predefined per run. The next sophistication is an orchestrator that right-sizes the team based on estimated task complexity — spawning fewer agents for small tasks and more for large ones.

  • 1Define a task complexity rubric: score incoming tasks on dimensions like scope (lines of code touched), dependency depth, and ambiguity in the PRD. A simple 1–5 score per dimension is enough to start.
  • 2Map complexity scores to team configurations: complexity 1–2 gets worker + reviewer only; 3–4 adds a docs agent; 5 spawns the full team including the analyzer. Hardcode this initially.
  • 3Once you have telemetry from gap 5, correlate complexity score with actual token cost and task duration. Validate whether your rubric predicts resource usage.
  • 4Replace the hardcoded mapping with an orchestrator LLM call that reasons about the task and outputs a team config. Use your eval suite to verify it doesn't over- or under-spawn.

Level up Self-improving prompts via synthetic data ▼

Currently you manually craft prompt improvements based on observed failures. The next level closes this loop automatically: the system generates candidate prompts, scores them, and surfaces the best ones for your review.

  • 1Build a synthetic task generator: given your PRD format and existing golden tasks, prompt an LLM to produce 50+ diverse new tasks. Review and curate a subset into your eval suite.
  • 2After each aggregated failure batch (from gap 4), use an LLM to draft 3 candidate prompt edits targeting the failure cluster. Store these as prompt candidates, not live prompts.
  • 3Run each candidate against your golden + synthetic eval set. Auto-promote any candidate that improves mean judge score by >0.2 points with no adversarial regression.
  • 4Explore DSPy's optimizer once you have 100+ eval cases — it can search the prompt space more systematically than manual iteration.

Level up Multi-model routing ▼

Every agent call likely hits the same model today. The next level routes by role and stakes — spending frontier-model budget only where it demonstrably improves output quality.

  • 1Use your cost telemetry (gap 5) to identify the two most expensive agent roles by token spend per run. These are your first routing candidates.
  • 2Run an experiment: replace the model on your lowest-stakes role (e.g. the docs agent) with a smaller/cheaper model. Score the output delta using your judge. If quality holds within 0.3 points, the swap is justified.
  • 3Build a routing config file that maps agent_role → model. Make it hot-swappable so you can change routing without touching agent code.
  • 4Add a "route by task complexity" layer: simple tasks on all roles get the cheaper model; tasks scoring 4–5 on your complexity rubric (gap 7) always get the frontier model for the worker role.

Level up Formal agent contracts and interface testing ▼

Agents currently communicate through files and prompts with no schema validation. A malformed handoff silently propagates rather than failing loudly at the boundary.

  • 1Define JSON schemas for each agent's expected output: what fields a worker must produce, what structure the reviewer consumes, what the docs agent emits. Start with just the worker → reviewer handoff.
  • 2Add a validation step between each agent handoff: parse the output against the schema and fail loudly (log to escape record + abort the loop) on any mismatch. No silent propagation.
  • 3Write contract tests for each agent role: given a known-good input, assert that the output matches the schema and passes a set of invariants (e.g. "reviewer output always contains a pass/fail verdict field").
  • 4Add the contract tests to your eval suite so any prompt change that breaks an output schema is caught before it ships.
