Start here Evaluation infrastructure ▼

Right now your only quality signal is "tests pass / fail." You have no way to know if agent output is improving, degrading, or whether your reviewer agent is actually doing its job.

  • 1Define 20–30 "golden tasks" — representative inputs with hand-written ideal outputs. Store them in evals/golden/ alongside your existing task format.
  • 2Write an LLM-as-judge prompt: score each agent output on a 1–5 rubric for correctness, completeness, and code quality. Run it after each agent cycle.
  • 3Log scores to evals/results/YYYY-MM-DD.jsonl — one line per task per run. This gives you a time series to track.
  • 4Set a baseline, then gate merges: if mean judge score drops below your baseline by >0.3 points, surface a warning before the reviewer agent signs off.

Start here Prompt iteration with feedback loops ▼

Your prompts improve only when you subjectively notice something's wrong. There's no mechanism to confirm a change is actually better.

  • 1Version your agent prompts as AGENT_PROMPT_v1.md, v2.md, etc. Never overwrite — always increment.
  • 2When you want to test a prompt change, run both versions against the same 10 golden tasks and compare judge scores. This is your A/B mechanism — no infrastructure needed beyond what you build in Gap 1.
  • 3Add a "prompt diff" section to your escape records: when an agent fails a task, note which prompt version was running. After 5–10 failures, look for version-correlated patterns.
  • 4Once you have 4+ prompt versions with score data, try DSPy or PromptBreeder to automate candidate generation — but only after you have a reliable judge to score candidates.

Layer in Agent memory architecture ▼

PROGRESS.md grows unbounded and every agent reads it from scratch each loop. Agents have no memory of prior attempts within a task run.

  • 1Split PROGRESS.md into a rolling summary file (last 5 items only) plus an archive. Agents read the summary; the archive is for humans and the improvement analyzer.
  • 2Add a per-task scratchpad file (.agent-state/task-{id}.md) that persists across agent loop iterations for that task. Agents can read their own prior attempts here before starting a new loop.
  • 3Define a trim policy: scratchpad files are deleted on task completion; the archive is summarized weekly by the analyzer agent.
  • 4Evaluate whether AthenaMind (or a similar vector store) actually helps vs the scratchpad approach — benchmark retrieval latency and token cost before committing.

Layer in Cross-agent failure aggregation ▼

Escape records are per-task and per-agent. You can't see patterns across agents — e.g., "three agents all failed on Databricks auth this week" only appears in hindsight.

  • 1Standardize your escape record schema: add agent_role, failure_category (auth, context, tool, logic), and task_domain fields to every entry.
  • 2Write a weekly aggregation script that groups escapes by failure_category × agent_role and outputs a Markdown table showing frequency. Run it as a cron or manually every Friday.
  • 3Upgrade the improvement analyzer to trigger not just on a single escape, but also on "3+ escapes with the same category in a 48h window." Add this pattern to its signal conditions.
  • 4Feed aggregated failure patterns back into the relevant agent prompts as an explicit "known failure modes" section — close the loop so agents pre-empt recurring errors.

Complete Cost and token observability ▼

At scale (10 agents × 8h runs), you have no accounting for which roles are expensive and whether expensive correlates with valuable output.

  • 1Wrap your RTK/API calls with a lightweight logger: capture agent_role, task_id, input_tokens, output_tokens, and timestamp per invocation. Append to telemetry/tokens.jsonl.
  • 2Build a daily rollup: cost per agent role, cost per task, and cost-per-judge-score-point (this is your ROI metric — did spending more tokens produce better output?)
  • 3Set soft budget alerts: if a single agent role exceeds a per-run token threshold (e.g. 200k input tokens), log a warning to PROGRESS.md so a human notices.
  • 4Use the data to make a concrete optimization decision: identify the highest-cost role and experiment with a cheaper model tier or a shorter context window. Measure the quality delta using your eval infrastructure.
