Who does what

What Each Role Reviews at Each Level

At each level, the artifact each role reviews changes. The work shifts upstream toward intent and downstream toward outcomes.

Product (PM) Development QA
L2
Now
 Writes user stories in ADO. Does manual acceptance testing by clicking through the product. Reviews PRs or asks devs for walkthroughs. Reviews every line of agent-generated code as it is produced. Owns architecture decisions entirely. Runs test suites manually. Triages every failure individually. Writes tests after implementation.
▼   L3 — Developer as Manager   ▼
L3
Step 1
 Reviews at the feature level — did it ship? does it solve the problem? No longer reviews every ticket or PR diff. Reviews at the PR level — reads the diff and test results instead of every individual commit. Spot-checks architecture. Reviews automated test results per feature. Triages suite-level failures vs individual test runs.
▼   L4 — Product Engineer   ▼
L4
Target
 Reviews acceptance verdicts — did each AC pass? Reads the report vs reading the code. Approves or triages failures by likely cause. Reviews architecture drift vs implementation. Maintains the test harness and handles agent escapes. Shifts toward platform reliability. Writes acceptance criteria and edge-case tests at spec time. Reviews AC coverage gaps and failure pattern trends. Doesn't need to review individual runs.

Day in the life — Starting now

A PM's Day at Level 3

Agents do the building; PMs review at the feature level. Manual acceptance testing shrinks significantly.

Morning
~45 min · active
  • → Write stories by hand in ADO, or describe in natural language and let a user story agent generate the ACs
  • → Check what agents completed overnight
  • → Review feature demos or PR descriptions for completed work
  • /spec-to-issues Create ADO items
  • /kickoff Agents start
Midday
passive
  • → Stakeholder calls, roadmap work
  • → ADO task progress visible on demand
  • → Occasional standup check-in still happens
Afternoon
~45 min · active
  • → Review PR descriptions and test results for completed features
  • → Spot-check a few ACs manually in the product
  • → Approve or request changes via ADO comment
  • → Agent may rerun if configured — but not guaranteed
End of day
~20 min · active
  • /sprint-report Draft sprint summary from closed ADO items — PM reviews and fills gaps
  • /plan-sprints Check backlog
When you're ready for Phase 2
Two things change when we move to L4. First, ACs get formally translated before work starts, removing the ambiguity that makes manual spot-checking necessary. Second, /accept generates a structured verdict for every AC, so the PM reads a report instead of reviewing the feature manually. Everything else — writing specs, kicking off agents, reviewing sprint health — carries over unchanged.

Day in the life — Level 4

A PM's Day at Level 4

This is where we're heading after L3 foundations are in place. Active PM time stays the same but afternoon review shifts from feature demos to reading acceptance reports.

Morning
~1 hour · active
Write specs & check overnight results
  • → Check ADO comments — see acceptance reports from last night's agent runs
  • → Triage any failures: revise spec, flag to dev, or accept as-is
  • → Describe the feature in natural language — ACs are auto-generated into a markdown spec file
  • /validate-spec Review auto-generated ACs, approve or edit, agent cannot start until approved
  • /spec-to-issues Create ADO work items from the spec
  • /kickoff Agents pick up tasks and start building
Midday
passive
Agents build — PM does other work
  • → Stakeholder calls, roadmap planning, user research
  • → ADO task-progress comments arrive automatically
  • /status Quick health check if needed — task progress, test results
Afternoon
~30 min · active
Review acceptance reports
  • → Acceptance report arrives in inbox — each AC passed or failed, with evidence
  • → All ACs pass → click Approve in ADO, feature ships
  • → Some ACs fail → read "likely cause", choose action:
  • · Ambiguous AC → revise spec, agent reruns
  • · Implementation gap → comment gets approved in ADO, agent re-runs against that AC
  • · Technical blocker → tag a dev in ADO
End of day
~15 min · active
Sprint health & stakeholder update
  • /sprint-report Auto-generated from acceptance reports — fully automated at L4, no gap-filling needed
  • /plan-sprints Check remaining backlog vs. agent velocity, adjust priorities
What you write

Three Types of Work Items

Each type has its own spec format and command pair. Bugs can be filed by anyone.

▶

Feature

Multi-story work with a defined goal. Full spec with user stories and acceptance criteria.

Author PM
File docs/specs/…feature.md
Commands /validate-spec → /accept
ACs Natural language, translated to verifiable form
■

User Story

A single scoped behavior. Lightweight with one goal, 1–3 acceptance criteria.

Author PM
File docs/specs/…story.md
Commands /validate-spec --story → /accept
ACs Same format, lighter validation
○

Bug

A reproduction case + expected vs. actual behavior. No acceptance criteria other than the bug no longer reproduces.

Author Anyone
File docs/bugs/…bug.md
Commands /bug-report → /verify-fix
ACs Implicit: repro no longer reproduces
New tools

Examples of New Commands

All global (available in any project).

Available at L3 — now
/bug-report
Run by: anyone

File a validated bug and create the ADO work item

Checks that the bug report has all required fields (steps to reproduce, actual vs. expected, severity). Creates the ADO Bug work item and writes the ID back to the file.

In: Bug report file
Out: ADO Bug work item created
/verify-fix
Run by: dev or agent after fix

Confirm a bug no longer reproduces before merging

Attempts to reproduce the bug against the current branch. Checks that a regression test was added. Reports VERIFIED or UNVERIFIED — a fix without a regression test is not accepted.

In: Bug report file + current branch
Out: Verification report in ADO + docs/acceptance/
/sprint-report
Run by: PM

Generate a stakeholder-ready sprint summary

Reads all completed work items and acceptance reports from the sprint window. Produces a plain-English summary of what shipped, what was blocked, and quality health metrics. When escape records use a standardized failure category schema, blocked items are grouped by pattern ("2 auth failures, 1 tool failure") rather than listed individually.

In: Sprint window (defaults to last 14 days)
Out: Summary doc + running metrics log
Unlock L4 — Phase 2
/validate-spec
Run by: PM

Translate acceptance criteria before agents start

Reads each natural-language AC and converts it to a specific, testable assertion. Shows a side-by-side. PM approves or edits each one. Agents cannot start until every AC is approved. This is the gate that makes structured acceptance review possible.

In: Your spec file
Out: Spec updated with verified ACs + approval timestamp
/accept
Run by: dev or agent after PR

Generate an acceptance report for PM review

Uses an LLM-as-judge to evaluate each verified AC against what was built — scores each as pass or fail with evidence. Posts a structured report as an ADO comment so the PM can approve without reviewing the PR.

In: Spec file + current branch
Out: Acceptance report in ADO + docs/acceptance/
How we know we're there

Graduation Metrics

L3 thresholds are the immediate target. When we sustain them for two consecutive sprints, we've graduated to L3. L4 thresholds follow once Phase 2 tooling is in place.

AC Pass Rate
Source: acceptance reports in docs/acceptance/
L3
> 70% of ACs pass
L4
> 90% sustained
Escape Rate
Source: ADO blocked items / tasks/escapes/
L3
< 20% of tasks escape
L4
< 10% of tasks escape
Regression Rate
Source: automated test suite (auto-detected per repo)
L3
< 10% regressions
L4
< 5% regressions
Critic Pass Rate
Source: docs/plans/*-critic-report.md
L3
> 70% critic checks pass
L4
> 85% critic checks pass
